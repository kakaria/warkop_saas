📘 BUKU SAKU ARSITEK: WARKOP SAAS (MULTI-TENANT)
FASE 1: Database Design & Logical Defense
Fokus utama: Merancang struktur data yang tidak mudah hancur oleh waktu dan kebal dari kebocoran antar warkop.The Tenant ID Rule: Di sistem SaaS Shared Database, setiap tabel transaksi (Order, Payment, Inventory) WAJIB memiliki kolom tenant_id. Tanpa ini, sistem isolasi runtuh.
Historical Data Immutability (Kekebalan Sejarah): Harga di tabel Product bisa berubah besok. Maka, di tabel OrderItem, harga saat itu (price_at_transaction) wajib disalin/di-copy. Jangan pernah menghitung total nota lama berdasarkan harga master yang baru.
Nullable Foreign Key (Produk Global vs Lokal): Untuk sistem yang menyediakan "Data Bawaan Pusat" sekaligus "Data Buatan User", buat kolom tenant_id menjadi Nullable (null=True, blank=True).
Jika tenant_id = NULL $\rightarrow$ Data Global (Pusat).
Jika tenant_id = 1 $\rightarrow$ Data Lokal (Milik Warkop 1).

The M2M Bridge (Tabel Jembatan): Jangan taruh Role/Jabatan di tabel User. Satu user bisa jadi Bos di Cabang A, tapi cuma jadi Kasir di Cabang B. Gunakan tabel jembatan TenantMembership untuk menghubungkan User, Tenant, dan Role.


FASE 2: Modular Architecture & App Decoupling
Fokus utama: Memecah aplikasi (Monolith) menjadi bagian-bagian kecil yang tidak saling mencekik.
One-Way Dependency: Aplikasi (apps) tidak boleh saling memanggil bolak-balik. Orders butuh data dari Products, tapi Products harus bisa hidup mandiri tanpa tahu menahu soal Orders.
Mencegah Circular Import (Lingkaran Setan): Di Django, jika App A me- import App B, dan App B me- import App A, server akan mati (crash). Solusinya: gunakan referensi string pada ForeignKey.❌ Salah: models.ForeignKey(Product, ...)✅ Benar: models.ForeignKey("products.Product", ...)
Identitas via Headers: Cara teraman mengenali Warkop mana yang sedang mengakses API adalah lewat HTTP Headers (misal: X-Tenant-ID), bukan lewat URL parameter yang mudah dimanipulasi orang iseng.


FASE 3: Sang Satpam (Middleware & Query Managers)
Fokus utama: Mengotomatisasi keamanan. Programmer gampang lupa, jadi sistem yang harus memaksa penyaringan data.
Bencana Concurrency & Thread-Local: Server menerima ribuan request bersamaan. Kalau tenant_id disimpan di variabel global, data Kasir A bisa tertukar dengan Kasir B. Solusinya: gunakan threading.local() (Loker Gym) agar setiap request punya memori terisolasi.
Middleware (Pendeteksi Pintu Depan): Kode yang berjalan sebelum masuk ke Views. Tugasnya mencegat Request, membaca KTP (Header X-Tenant-ID), menyimpannya di Loker Gym, dan wajib membersihkannya (clear) di blok finally setelah request selesai.
Custom Query Manager (Eksekutor Gaib): Meng-override (menimpa) fungsi get_queryset() bawaan Django. Dia akan membaca ID dari Loker Gym dan otomatis menyuntikkan .filter(tenant_id=...). Jadi saat lu nulis Order.objects.all(), yang tereksekusi di database sebenarnya adalah SELECT * FROM orders WHERE tenant_id = 1.
The Bypass Mechanism: Selalu sediakan manajer cadangan seperti global_objects = models.Manager() untuk Superadmin yang butuh menarik semua data tanpa filter Satpam.Q Objects (Logika OR): Untuk membaca Produk Global (NULL) dan Produk Lokal bersamaan, gunakan Q(tenant_id=1) | Q(tenant__isnull=True).


FASE 4: Async Engines (Celery & Background Jobs)
Fokus utama: Mencegah server mati (timeout) saat memproses tugas berat seperti ekspor PDF laporan tahunan.Context Loss: Celery berjalan di terminal belakang (background), dia tidak dilewati oleh Middleware. Maka, tenant_id wajib dioper sebagai parameter ke dalam fungsi Celery (@shared_task), lalu dimasukkan kembali ke Loker Gym secara manual di dalam fungsi tersebut.
State Machine (Pencegah Tombol Panik): User sering spam klik tombol "Buat Laporan". Cegah server kehabisan RAM (OOM) dengan membuat tabel Report yang punya kolom status (PENDING, GENERATING, SUCCESS, FAILED).
Idempotency Flow: Sebelum melempar tugas ke Celery, cek database: Jika warkop ini masih punya laporan berstatus PENDING/GENERATING, tolak request-nya.
Polling: Frontend tidak perlu menunggu respons Celery (karena koneksi akan terputus). Frontend cukup bertanya (Polling) ke Backend setiap 5 detik: "Laporan ID 10 udah SUCCESS belum?"


FASE 5: Automated Testing (TDD & Data Isolation Check)
Fokus utama: Membuktikan secara matematis bahwa sistem tidak bisa dibobol atau bocor (Data Leak).
Setup Lingkungan Uji: Di fungsi setUp(), buat Tenant A dan B, buat pesanan untuk A dan B menggunakan pintu belakang (global_objects), lalu loginkan Kasir A.
Memanipulasi Request Palsu: Saat menembak API di lingkungan tes (DRF APIClient), tambahkan prefix HTTP_ pada header. Contoh: HTTP_X_TENANT_ID=1.
Validasi Mutlak (Assertions): Gunakan assertEqual untuk memastikan jumlah data sesuai, dan assertNotIn untuk memastikan mutlak bahwa ID Pesanan B tidak pernah muncul di layar Kasir A.
Jebakan Import Time vs Runtime: Di Django REST Framework (Generic Views), jangan menaruh queryset = Order.objects.all() sebagai variabel statis di tingkat Class. Itu akan dievaluasi saat server nyala (sebelum ada request), membuat Manajer memblokir data secara permanen. 
Solusinya: Gunakan fungsi dinamis def get_queryset(self): return Order.objects.all() agar Manajer berjalan tepat di detik request masuk (Runtime).
